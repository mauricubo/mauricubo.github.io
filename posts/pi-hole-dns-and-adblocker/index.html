Pi-Hole - DNS and AdBlocker

Oct 24, 2022 · 851 words · 4 minute read

This application was the first one that I wanted to try when I installed and configured my personal server. As I wrote in my last post, I’ve installed and configured a personal server at home to run some applications for my personal use, so my next step was this really popular application pi-hole.

pi-hole is an opensource and free software that offers a DNS, DHCP and an incredible Ad-blocker (it has other additional functions but the ones that I mention are the core). So let’s get going and install and configure our network.

The softwares that I use to run pi-hole are:

  1. Ubuntu - as my operating system
  2. microk8s - as my kubernetes manager
  3. Metallb - as my LB
  4. nginx - as my ingress controller
  5. Pi-Hole

💡 Note: For the steps 1, 2, 3 and 4 you can follow the link below.

My Home Server

  1. We are going to create our deployment file that contains the namespace and the specs for the pod that we are going to run. We basically are adapting the docker-compose way to run this application into a kubernetes way.

If we look for the docker-compose command we can find something like:

version: "3"

# More info at https://github.com/pi-hole/docker-pi-hole/ and https://docs.pi-hole.net/
services:
    pihole:
    container_name: pihole
    image: pihole/pihole:latest
    # For DHCP it is recommended to remove these ports and instead add: network_mode: "host"
    ports:
        - "53:53/tcp"
        - "53:53/udp"
        - "67:67/udp" # Only required if you are using Pi-hole as your DHCP server
        - "80:80/tcp"
    environment:
        TZ: 'America/Chicago'
        # WEBPASSWORD: 'set a secure password here or it will be random'
    # Volumes store your data between container upgrades
    volumes:
        - './etc-pihole:/etc/pihole'
        - './etc-dnsmasq.d:/etc/dnsmasq.d'
    #   https://github.com/pi-hole/docker-pi-hole#note-on-capabilities
    cap_add:
        - NET_ADMIN # Required if you are using Pi-hole as your DHCP server, else not needed
    restart: unless-stopped

In summary, this is the information we can find:

  • Name of the container and image
  • Ports that are mapped with the host
  • Environment variables
  • Volumes to persist the container data
  • cap_add - we are not going to use it because I’m not using it as my DHCP server

If we translate the docker-compose.yaml file into a deployment.yaml we are getting something like:

apiVersion: v1
kind: Namespace
metadata:
    name: pihole
---
apiVersion: apps/v1
kind: Deployment
metadata:
    name: pihole
    namespace: pihole
spec:
    selector:
    matchLabels:
        app: pihole
    template:
    metadata:
        labels:
        app: pihole
    spec:
        containers:
        - name: pihole
        image: pihole/pihole:latest
        ports:
        - containerPort: 80
            name: piholeweb
            protocol: TCP
        - containerPort: 53
            name: piholednsudp
            protocol: UDP
        - containerPort: 53
            name: piholednstcp
            protocol: TCP
        env:
            - name: 'TZ'
            value: 'America/Lima'
        volumeMounts:
            - mountPath: '/etc/pihole'
            name: pihole-config
            - mountPath: '/etc/dnsmasq.d'
            name: dnsmasqd-config
        volumes:
        - name: pihole-config
            hostPath:
            path: /app/kubernetes/pihole/etc/pihole
        - name: dnsmasqd-config
            hostPath:
            path: /app/kubernetes/pihole/etc/dnsmasq.d

That’s for the pod deployment part, but now we need to add the exposition of the service or the port and IP mapping to the network.

As I’ve mentioned before, we have configured Metallb as our Load Balancer, so now we are going to assign an specific IP from the network to expose the DNS service ports and expose it to the Web Interface using our NGINX ingress controller.

Our DNS service exposition is going to be something like this.

apiVersion: v1
kind: Service
metadata:
    name: dns-service
    namespace: pihole
    annotations:
    metallb.universe.tf/address-pool: default-addresspool
spec:
    type: LoadBalancer
    loadBalancerIP: 192.168.5.50
    ports:
    - name: dnstcp
        protocol: TCP
        port: 53
        targetPort: 53
    - name: dnsudp
        protocol: UDP
        port: 53
        targetPort: 53
    selector:
    app: pihole

And our Web Interface exposition is going to be like this:

apiVersion: v1
kind: Service
metadata:
    name: piholeservice-web
    namespace: pihole
spec:
    selector:
    app: pihole
    ports:
    - port: 80
    targetPort: piholeweb
    protocol: TCP
---
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
    name: home-ingress
    namespace: pihole
    labels:
    app: home-ingress
    annotations:
    #nginx.ingress.kubernetes.io/rewrite-target: /
    nginx.ingress.kubernetes.io/app-root: /admin
spec:
    ingressClassName: nginx
    rules:
    - host: "pihole.mauricubo.com"
    http:
        paths:
        - pathType: Prefix
        path: /
        backend:
            service:
            name: piholeservice-web
            port:
                number: 80

You can notice that the DNS service exposition is only a service that maps an IP from the Metallb service (LoadBalancer Type), but our Web Interface exposition is using the internal Service (ClusterIP Type) and then configures the Ingress that is exposed to the NGINX Ingress Controller.

It’s good to remember this, because it’s going to be a standard way to deploy our applications on our cluster. Most of the apps that we are going to install need a deployment, a service and an ingress to be exposed to the network.

Now we can apply our configuration files.

$ kubectl apply -f deployment.yaml
namespace/pihole created
deployment.apps/pihole created

$ kubectl apply -f dns_service.yaml
service/dns-service created

$ kubectl apply -f web_service.yaml
ingress.networking.k8s.io/home-ingress created

We can test it now…
