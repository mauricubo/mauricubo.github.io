Deploy your own hosted Ghost Blog on Kubernetes - Part 1

Sep 19, 2022 · 1647 words · 8 minute read

I’ve started to use Ghost as a platform to write my nerdy post, because I would like to use it as my personal notebook, but I want to use it in a good way to contribute to the community as well (because everything is possible googling). But this time, I decided to host my own blog and do all the work from scratch and I want to share what I’m learning while doing it.

Here is the list of the technologies and tools that I used to implement this blog and I’m going to explain how to integrate it step by step.

  • Ghost Blog
  • DigitalOcean
  • Docker
  • Kubernetes
  • Letsencrypt
  • MySQL

For a better understanding what we are going to implement, it’s always good to start with a high level architecture diagram like this one:

Step 1 - Create your DigitalOcean account and Kubernetes Cluster 🔗

This tutorial was made using a DigitalOcean account. If you are a DO rookie user they offer you a $100 coupon that you can spend in 60 days (we are going to need much less than that for this tutorial). I’ve used this cloud for personal projects and it’s really good, additionally that it has a good integration with other tools like terraform, ansible, etc. Here you have a referral coupon to create your account and you are helping me with $25 for my personal use.

Once that you create your DO account and have access to it, we are going to create our kubernetes cluster. You can do it manually, using terraform, ansible or the DO cli.

  1. Go to Kubernetes and then click the Create Cluster button.

  1. Select the region, in my case I selected NYC1. Select the version, it’s recommended that you select the latest one. Give a name to your nodes pool and select how many nodes you want to add to your cluster (remember the more you add, the more that you are going to pay for it).

  1. Select a name for your cluster and the Project (it’s like a workspace) and finally click the Create Cluster button.

  1. Once your cluster is created, you need to add the configuration to the kubectl tool to manage it.

  1. If you have no config files on your ~/.kube folder you can create it and copy the config file to that directory, but if you already have the ~/.kube/config file we need to merge those files to add the context to the kubectl context list. In that case, we need to extract these three things from the context file that we recently downloaded.

Those things are cluster, context and user

apiVersion: v1
clusters:
# copy this part
- cluster:
    certificate-authority-data: <cert_data>
    server: <server_url>
name: <cluster_name>
# end of the first part
contexts:
# copy this part
- context:
    cluster: <cluster_name>
    user: <user_name>
name: <context_name>
# end of the second part
kind: Config
preferences: {}
users:
# copy this part
- name: <user_name>
user:
    token: <user_token>
# end of the third part
  1. Switch context to connect to the new cluster that we already create. I use the kubectx tool to easily switch between my context, like this:
$ kubectx
do-nyc1-k8s-mauricubo-cluster
do-nyc1-k8s-mycluster
docker-desktop

and we can switch like this

$ kubectx do-nyc1-k8s-mauricubo-cluster
Switched to context "do-nyc1-k8s-mauricubo-cluster".

now that we are connected we can control our cluster with kubectl

$ kubectl get nodes
NAME                       STATUS   ROLES    AGE   VERSION
k8s-mauricubo-pool-7gb9f   Ready    <none>   84m   v1.24.4
k8s-mauricubo-pool-7gb9q   Ready    <none>   84m   v1.24.4

💡 Note: It’s a good practice to create different namespaces for our projects. In this case we are going to create our ghost-prod namespace like this: $ kubectl create namespace ghost-prod.

Step 2 - Create the configuration files and objects in Kubernetes 🔗

As we can see in our architecture diagram, we have a few components that we need to deploy on our cluster, and some of them represents more than one object in our Kubernetes cluster, so, let start deploying our MySQL service.

Deploying the MySQL Service 🔗

  1. Create a Secret object to store our MySQL password into our Kubernetes, it’s not the most secure option but it’s a better practice than setting it into our application code. We are going to name it mysql-secret.yaml
apiVersion: v1
kind: Secret
metadata:
    name: mysql-ghost-secrets
    namespace: ghost-prod
type: Opaque
stringData:
    password: <change_this_password>
  1. Create Deployment and Service objects that are going to manage all the MySQL implementation, like the pod, storage, service, etc. This file is going to be the ghost_mysql_deployment.yaml
apiVersion: v1
kind: Service
metadata:
    name: ghost-mysql
    namespace: ghost-prod
    labels:
    app: ghost
spec:
    ports:
    - port: 3306
    selector:
    app: ghost
    tier: mysql
    clusterIP: None
---
# Change depends of the cloud or service
# that you are using for kubernetes
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
    name: ghost-prod-mysql-pvc
    namespace: ghost-prod
spec:
    accessModes:
        - ReadWriteOnce
    resources:
    requests:
        storage: 20Gi
    storageClassName: do-block-storage
---
apiVersion: apps/v1
kind: Deployment
metadata:
    name: ghost-mysql
    namespace: ghost-prod
    labels:
    app: ghost
spec:
    selector:
    matchLabels:
        app: ghost
        tier: mysql
    strategy:
    type: Recreate
    template:
    metadata:
        labels:
        app: ghost
        tier: mysql
    spec:
        containers:
        - image: mysql:8.0.30
        name: ghost-mysql
        env:
        - name: MYSQL_ROOT_PASSWORD
            valueFrom:
            secretKeyRef:
                name: mysql-ghost-secrets
                key: password
        ports:
        - containerPort: 3306
            name: mysql
        volumeMounts:
        - name: mysql-persistent-storage
            mountPath: /var/lib/mysql
        volumes:
        - name: mysql-persistent-storage
        persistentVolumeClaim:
            claimName: ghost-prod-mysql-pvc

Before applying this configuration file to our Kubernetes cluster, let’s take a look at some important objects in the file to understand what we are doing:

  • Service: It’s the object that describes how to access to the application, in this particular case, we are indicating that the service is exposed to the TCP port 3306 with the label app: ghost.
  • Deployment: As the official documentation says, provides declarative updates for the pods, wich means that define the desired state of the pods.
  • Secret: Link the secret object that we have already created with the pod in order to access to the MySQL password in this case.
  • Persistent Volume Claim: These are request to access and assign Persistent Volumes, in simple words, we are assinging storage to the pod and mounting the /var/lib/mysql to the volume. This type of objects depends on the type of storage that we are assigning to the pod, in this case I’m using the DigitaOcean storage. That’s why I’m using storageClassName: do-block-storage, if you want to work with a different storageClass, you should take a closer look at the documentation
  1. Apply the configuration into the Kubernetes Cluster
$ kubectl apply -f mysql-secret.yaml
$ kubectl apply -f ghost_mysql_deployment.yaml

And we now can see that the MySQL service is up and running:

$ kubectl get pods -n ghost-prod    
NAME                           READY   STATUS    RESTARTS   AGE
ghost-mysql-5c685d9d7b-sz9dn   1/1     Running   0          2m18s
$ kubectl get services -n ghost-prod
NAME          TYPE        CLUSTER-IP   EXTERNAL-IP   PORT(S)    AGE
ghost-mysql   ClusterIP   None         <none>        3306/TCP   2m25s

Deploying the Ghost Blog Service 🔗

  1. Now we are going to do something similar but with the ghost blog service. We are going to use the docker image that is in the official repository (here the link). The file is named ghost_deployment.yaml
apiVersion: v1
kind: Service
metadata:
    name: ghost-blog
    namespace: ghost-prod
spec:
    selector:
    app: ghost-blog
    ports:
    - port: 80
    targetPort: 2368
    protocol: TCP
---
# Change depends of the cloud or service
# that you are using for kubernetes
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
    name: ghost-blog-pvc
    namespace: ghost-prod
spec:
    accessModes:
        - ReadWriteOnce
    resources:
    requests:
        storage: 10Gi
    storageClassName: do-block-storage
---
apiVersion: apps/v1
kind: Deployment
metadata:
    name: ghost-blog
    namespace: ghost-prod
    labels:
    app: ghost-blog
    tier: web
spec:
    replicas: 1
    selector:
    matchLabels:
        app: ghost-blog
    template:
    metadata:
        labels:
        app: ghost-blog
    spec:
        # https://hub.docker.com/_/ghost/
        containers:
        - name: ghost-blog
        image: ghost:latest
        imagePullPolicy: Always
        ports:
        - containerPort: 2368
        env:
            - name: url
            value: https://blog.mauricubo.com
            - name: database__client
            value: mysql 
            - name: database__connection__host
            value: ghost-mysql
            - name: database__connection__user
            value: root
            - name: database__connection__password
            valueFrom:
                secretKeyRef:
                name: mysql-ghost-secrets
                key: password
            - name: database__connection__database
            value: ghost
        volumeMounts:
        - mountPath: /var/lib/ghost/content
            name: blog-content
        volumes:
        - name: blog-content
        persistentVolumeClaim:
            claimName: ghost-blog-pvc
  1. We run the commands to apply the config files to the cluster
$ kubectl apply -f ghost_deployment.yaml

And we check if everything is running as expected.

$ kubectl get pods

ghost-mysql-5c685d9d7b-sz9dn   1/1     Running   0          17m
$ kubectl get services
NAME          TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)    AGE

Step 3 - Publish the blog to internet 🔗

Now we have all our basic services configured on the Kubernetes cluster, the app is working now but is not exposed to the internet, so, the final users still doesn’t have access to our future blog.  Let’s do the final step from this post.

  1. Let’s create our NGINX Ingress Controller who is responsible to balance all the external traffic and distribute internally to our service that are running in our node pools. To know more about the service, the official documentation has a good explanation about the service and it function itself.

In order to create this service, we are going to execute an external configuration file that I took following this tutorial.

$ kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/controller-v1.3.1/deploy/static/provider/do/deploy.yaml

You should notice that the execution of that configuration is going to create an external LoadBalancer, it is integrated with the DigitalOcean service and the cloud itself is going to assignate an external public IP that is the service where your service is going to be exposed.

  1. Our next step is linking the ingress controller with the ghost blog service in order to expose it to the internet and assign it a real domain name. Let’s call it ghost_ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
    name: ghost-blog-ingress
    namespace: ghost-prod
    annotations:
    kubernetes.io/ingress.class: "nginx"
    cert-manager.io/cluster-issuer: "letsencrypt-prod"
spec:
    rules:
    - host: mypersonalblog.mauricubo.com
    http:
        paths:
        - pathType: Prefix
        path: "/"
        backend:
            service:
            name: ghost-blog
            port: 
                number: 80

$ kubectl apply -f ghost_ingress.yaml
$ kubectl get ingress
NAME                 CLASS    HOSTS                          ADDRESS         PORTS   AGE
ghost-blog-ingress   <none>   mypersonalblog.mauricubo.com   24.199.66.173   80      5m27s
  1. Now, we only need to add the domain name point to that public IP address.

And finally we have published our blog to internet: 🔗
