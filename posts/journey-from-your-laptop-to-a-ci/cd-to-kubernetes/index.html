Journey from your laptop to a CI/CD to Kubernetes

Feb 10, 2023 · 1590 words · 8 minute read

In today’s post I want to write something that came to my mind when I was trying to write code. In particular I was trying to remember how to write code again, so I started to write some kind of API with Flask.

Many people love Flask for its simplicity and I wanted to give it a try to see how easy it was to create a functional API from scratch. I’m going to talk about my experience later, but what I would like to remark now, was my instant desire to automate the deployment process. I didn’t start to code and I was thinking to create a CI/CD pipeline to automate the whole DevSecOps process that is part of our daily basis as DevOps fans.

So, I started to think how the process was going to be, from starting with my code on my local computer running with the development tools that comes by default on our environment, to deploying the app to a Kubernetes cluster through a pipeline that controls all the software’s lifecycle. For that reason I wrote the steps that I wanted to follow in order to respect the whole process and focus on what I was trying to do.

  1. Create a full stack application with base on Python+Flask as my backend and React as my frontend.
  2. Create the Dockerfiles and interconnect the projects
  3. Use docker-compose to orchestrate and link my project
  4. Deploy the app in a Kubernetes Cluster
  5. Standardize the deployment using Helm
  6. Create a pipeline
    • Build the docker images
    • Deploy the images on kubernetes
    • Add vulnerability scans and unit tests
    • Connect with other platforms

The Url Shortener App 🔗

The Url Shortener Project is an application that generates randoms and shorts urls and redirect the clients to the final site. It’s exactly like the google url shortener or bitly. I just wanted to create an API Rest with Flask so that idea came to my mind.

This is the UML Diagram to understand the model.

It’s a user that can have as many urls as it wants. And my apidocs looks like this:

Now I can do the following things:

  1. Register and authenticate users
  2. Get my user info details
  3. Create, read and delete items from my URL list

From my perspective, create an API from scratch in Flask was a quite interesting experience, and when I say experience I’m talking about you starting from scratch and finally creating a monster.

I literally have started with this example:

from flask import Flask

app = Flask(__name__)

@app.route("/")
def hello_world():
    return "Hello, World!"

But now I have a complete new project, of which I needed to rethink the structure and which was growing to a more mature structure, but which could still be improved to be more organized.

I could see as well that many people love that facility and freedom to create their project and structure as the way that they want, but many others don’t like that, and prefer other frameworks, like django. I could say that the structure that I have right now was growing in a natural way, kind of like I was guided to that structure during the process when I was implementing more functions to my project. If I have to describe what it looks like, I have to say that the structure now looks similar to django.

Containers and Configs 🔗

But ok, I don’t want to continue talking about coding, now I want to talk about how to pass from running the default development server to run the apps on docker container.

A simple way to start with containers is to run the applications on a single instance of an image specially prepared for that propose.

Flask App - Backend 🔗

FROM python:3.9.2-slim as flask

RUN pip install --upgrade pip

COPY --chown=www-data:www-data . /usr/app/url_shortener
RUN chown -R www-data:www-data /usr/app/url_shortener
WORKDIR /usr/app/url_shortener

RUN apt-get clean \
    && apt-get -y update
RUN apt-get -y install nginx \
    && apt-get -y install python3-dev \
    && apt-get -y install build-essential \
    && apt-get -y install libpq-dev

RUN pip install -r requirements.txt
RUN mv nginx/nginx.conf /etc/nginx
RUN chmod +x nginx/startup.sh

CMD ["./nginx/startup.sh"]

🐋 Note: I need to reduce the image size of the flask app.

Because we are going to use uwsgi to run our app, we need to configure the server and expose it behind the nginx server. That’s why we need the uwsgi.ini, the nginx.conf and the startup.sh

[uwsgi]
module = main:app
uid = www-data
gid = www-data
master = true
processes = 5

socket = /tmp/uwsgi.socket
chmod-sock = 664
vacuum = true

die-on-term = true

uwsgi does not expose the app as standard http protocol, that’s why we need the nginx server.

user www-data;
worker_processes auto;
pid /run/nginx.pid;

events {
    worker_connections 1024;
    use epoll;
    multi_accept on;
}

http {
    access_log /dev/stdout;
    error_log /dev/stdout;

    sendfile            on;
    tcp_nopush          on;
    tcp_nodelay         on;
    keepalive_timeout   65;
    types_hash_max_size 2048;

    include             /etc/nginx/mime.types;
    default_type        application/octet-stream;

    index   index.html index.htm;

    server {
        listen       80 default_server;
        listen       [::]:80 default_server;
        server_name  localhost;
        root         /var/www/html;

        location / {
            include uwsgi_params;
            uwsgi_pass unix:/tmp/uwsgi.socket;
        }
    }
}

The start script to run nginx and the uwsgi service.

#!/usr/bin/env bash
service nginx start
uwsgi --ini uwsgi.ini

Now we just build the image…

$ docker build --progress=plain --no-cache --platform linux/amd64 . --tag url_shortener_be:latest

🤯 Note: --progress=plain --no-cache is to show and persist the progress in the terminal. --platform linux/amd64 is a special parameter that I had to use to build a compatible image with my linux system when I used MacOS.

And run it like this.

$ docker run -it -p 8002:80 -e CONFIGURATION_SETUP="config.DevConfig" url_shortener_be:latest

Note that I’m using an environment variable to set the config on running time. It’s because the image and the code are the same for any environment but the config is not.

This is my config.py file that manages which config is loading in the app depending on the environment variable.

    from os import environ
    
    class Config:
        """Base config"""
        DEBUG = False
        TESTING = False
        SECRET_KEY = environ.get('SECRET_KEY')
        JWT_SECRET_KEY = environ.get('JWT_SECRET_KEY')
        SHORT_BASE_URL = "https://ma.al/"
        SQLALCHEMY_TRACK_MODIFICATIONS = False
        BCRYPT_LOG_ROUNDS = 10
        SWAGGER = {
            'title': 'URL Shortener API',
            'uiversion': 3
        }
    
    
    class ProductionConfig(Config):
        # DATABASE CONFIG
        DB_SERVER = environ.get('DB_SERVER')
        DB_TYPE = 'postgresql'
        DB_USERNAME = environ.get('DB_USERNAME')
        DB_PASSWORD = environ.get('DB_PASSWORD')
        DB_NAME = environ.get('DB_NAME')
        SHORT_BASE_URL = environ.get('SHORT_BASE_URL')
        SQLALCHEMY_DATABASE_URI = f"{DB_TYPE}://{DB_USERNAME}:{DB_PASSWORD}@{DB_SERVER}/{DB_NAME}"
    
    
    
    class DevConfig(Config):
        ENV = 'development'
        TESTING = False
        DEBUG = False
        SECRET_KEY = '<dev_secret_key>'
        JWT_SECRET_KEY = '<dev_jwt_secret_key>'
        # DATABASE CONFIG
        SQLALCHEMY_DATABASE_URI = 'sqlite:///urls.db'
        SHORT_BASE_URL = "https://dev.ma.al/"

You can add as many configs as you want, in this particular case the big difference is that I’m using an sqlite instance for my dev environment and a postgresql instance for my production environment.

React App - Frontend 🔗

This part of the app was not so easy to start without having some knowledge about React. It really took me some time to understand how to deal with some basics things and debuging the app as well. But I finally did it, I wrote some functional frontend app (that really needs more love, but it’s working). So let’s see how we can change from npm start to run the app inside a docker container.

FROM node:current-alpine3.17 AS build
WORKDIR /usr/app
COPY . /usr/app/
ARG REACT_APP_BASE_API_URL
ENV REACT_APP_BASE_API_URL $REACT_APP_BASE_API_URL
RUN npm ci
RUN npm run build

FROM nginx:alpine-slim
EXPOSE 80
COPY ./nginx/default.conf /etc/nginx/conf.d/default.conf
COPY --from=build /usr/app/build /usr/share/nginx/html

There is no mystery here, but the real problem, I have to say, was using environment variables inside the project. I just only wanted to pass an environment variable as the base url of the API, but I found out that React is really picky when working with envs, and even more when it runs into a Docker container.

The only way that I found was to build the image and pass the environment variable while building it. To be more clear, this is how the docker part looks when you set the environment variable.

ARG REACT_APP_BASE_API_URL
ENV REACT_APP_BASE_API_URL $REACT_APP_BASE_API_URL

And this is the docker build command that I had to use to build the image.

$ docker build --progress=plain --no-cache --platform linux/amd64 . --tag url_shortener_fe:latest --build-arg REACT_APP_BASE_API_URL="http://localhost:8002/api/v1/"

🙃 Note: The --build-arg parameter was the only easy way without touching the code that I found to work with envs

And finally it’s running like this…

$ docker run -it -p 8003:80 url_shortener_fe:latest -e REACT_APP_BASE_API_URL='http://localhost:8002/api/v1/'

My app is up and running.

Docker Compose 🔗

Last but not least, let’s run everything together with docker-compose. I’m only going to change the database engine to be prepared when we think to deploy our app in a different place than our laptop.

Our docker-compose.yaml looks like this…

version: "3.9"
services:
    web:
    build: ./url_shortener_be
    ports: 
        - "8000:80"
    environment:
        FLASK_ENV: "production"
        SECRET_KEY: "<secret_key>"
        JWT_SECRET_KEY: "<jwt_secret_key>"
        DB_SERVER: "db"
        DB_TYPE: "postgresql"
        DB_USERNAME: "postgres"
        DB_PASSWORD: "Passw0rd123"
        DB_NAME: "url_shortener"
        SHORT_BASE_URL: "http://ma.alp/"
        CONFIGURATION_SETUP: "config.ProductionConfig"
        DOCKER_DEFAULT_PLATFORM: "linux/amd64"
    depends_on:
        - db
        - frontend

    db:
    image: postgres
    environment:
        POSTGRES_PASSWORD: "Passw0rd123"
        POSTGRES_DB: "url_shortener"
    ports:
        - "5432"

    frontend:
    build: 
        context: ./url_shortener_fe
        args:
        REACT_APP_BASE_API_URL: "http://web:8000/api/v1/"
    ports:
        - "3000:80"
    depends_on:
        - db

ℹ️ Note: Notice that we are building our docker-compose.yaml file from a different folder level that where we have our Dockerfiles.

ℹ️ Note: The second thing to notice is that we add the db configuration on the compose file and we link the configuration in the api configuration.

Now, we just simple run…

$ docker-compose build .
$ docker-compose up -d 
[+] Running 3/3
    ⠿ Container url_shortener-db-1        Started                                                                                          0.3s
    ⠿ Container url_shortener-frontend-1  Started                                                                                          0.5s
    ⠿ Container url_shortener-web-1       Started                                                                                          0.7s

And now the app is running on our local machine but in 3 different containers and ready to be used on another type of deployment.
