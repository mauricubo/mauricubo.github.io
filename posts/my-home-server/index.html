My Home Server

Oct 24, 2022 · 1540 words · 8 minute read

I finally decided to buy a mini pc to start playing with some basics things that we don’t really need but that are really helpful when you are surfing the web from home. The main reason for doing that was because I’m thinking to move my personal blog to my own hosting, MY HOME SERVER.

Technology/Software that I have used:

  1. Mini Pc
  2. Ubuntu Server 22.04
  3. Microk8s
  4. Metallb
  5. Heimdall Application Dashboard

Here is a diagram of how my home server network and components are working right now.

Actually I’m using a dedicated firewall, but the commercial routers/modems already have all the functions.

So… Let’s start

The Mini PC 🔗

I did some research about mini pcs and the raspberry pi (I was really tempted about a raspberry pi cluster idea. link here) looking and comparing prices and models available. Finally I decided to buy the CHUWI LarkBox Pro Mini PC to start the game. After testing it for a while I’ll decide if I move to another solution or if I continue with this line (buying more of this mini pcs).

I’m going to show you a real picture so you can see the dimensions of this small thing.

That little thing comes with 6Gb RAM, 128Gb SSD, Intel® Celeron® J4125 Quad Core, and Windows 10 installed by default and runs perfectly on that hardware, of course we are not going to using it as the operating system for a personal server, but I think that this small thing can be used as a personal computer without any problems. There’s only one bad thing about this PC, and I think that’s because it’s designed to be small, but it doesn’t have an Ethernet Port to connect to the network. If you want to connect the Mini PC directly to the switch you have to think about buying a USB adapter or use it while connected to the Wifi (supports 2.4G and 5G), which works perfectly.

The Operating System 🔗

I didn’t want to complicate the things too much, because I was thinking to install microk8s, I decided to install Ubuntu 22.04 as the server operating system. It’s a really good operating system, I feel comfortable with Ubuntu and I didn’t need so much customization to run Kubernetes. I’m not going to explain the step-by-step installation, but I basically copied the image into a pendrive and modified the /etc/netplan/00-installer-config-wifi.yaml

# cat /etc/netplan/00-installer-config-wifi.yaml
# This is the network config written by 'subiquity'
network:
  version: 2
  wifis:
    wlan0:
      access-points:
        <your_wifi_network_name_here>:
          password: <your_wifi_password_here>
      dhcp4: false
      addresses: [<the_minipc_ip_address/24]
      nameservers:
          addresses: [8.8.8.8,8.8.4.4]
      routes:
          - to: default # could be 0/0 or 0.0.0.0/0 optionally
            via: <your_default_gw>
            metric: 100
            on-link: true

Microk8s 🔗

It’s an opensource software to manage the kubernetes installation and administration (you can see the kubeadm way in my other post here). It’s really simple to install and wrap the installation of many add-ons so you can easily install or uninstall these features in the future without doing it from scratch. It’s a tool that you can use for small production environments.

MicroK8s - Zero-ops Kubernetes for developers, edge and IoT | MicroK8s

We install microk8s service on the server

$ sudo snap install microk8s --classic
microk8s (1.25/stable) v1.25.3 from Canonical✓ installed

We add the user to the microk8s group to manage the kubernetes

$ sudo usermod -a -G microk8s $USER
$ sudo chown -f -R $USER ~/.kube
$ su - $USER
Password:

We check the status of the installation

Create an alias to kubectl and configure the autocompletion

$ alias kubectl='microk8s kubectl'
$ echo 'source <(kubectl completion bash)' >>~/.bashrc
$ echo 'alias k=kubectl' >>~/.bashrc
$ echo 'complete -o default -F __start_kubectl k' >>~/.bashrc

Everything is working perfectly…

Ok, now we have a single node cluster of kubernetes managed by the microk8s, we can add or remove more nodes only with the join command. But that’s material for a future post…. for now I’m using one of the mini pcs only to see how it is working as a personal server.

Metallb - The Bare Metal Load Balancer 🔗

When we install kubernetes on Bare Metal solutions (means that you are using your own infrastructure, it’s dedicated, different like cloud environments for example), we don’t have a default load balancer to redirect the traffic between the nodes on a same network. Most of the cloud providers have a default integration with this part of the cluster that allows you to configure the ingress and create a load balancer that points to your nodes of the cluster.

Now we are going to configure the metallb with the layer2 config, it’s not the best configuration for a node failure but it’s the easiest configuration and it’s going to work anyway but it’s going to take longer to pass the IP address to the active node. In my case, as I have just one node, it’s not going to be a problem. This is our metallb.yaml configuration to set the IP range in the kubernetes config. I’m using the IPs of my local network, the same that I assign to the host to be part of the network.

---
apiVersion: metallb.io/v1beta1
kind: IPAddressPool
metadata:
  name: default-addresspool
  namespace: metallb-system
spec:
  addresses:
  - 10.116.0.10-10.116.0.15
  autoAssign: true
---
apiVersion: metallb.io/v1beta1
kind: L2Advertisement
metadata:
  name: default
  namespace: metallb-system
spec:
  ipAddressPools:
  - default

We apply the configuration.

$ kubectl apply -f metallb.yaml
ipaddresspool.metallb.io/default-addresspool configured
l2advertisement.metallb.io/default created

Heimdall Application Dashboard 🔗

Now that we have our cluster ready, lets install our first application. There are many good projects on github ready to be installed and free to use that it was hard to chose with which one I should start. I finally decided to start with a dashboard, an application with which I can centralize my existing applications and link the future ones to.

The Heimdall Application Dashboard makes managing your websites and applications much easier.

So, let’s take a look at the steps that we need to deploy the app on our new cluster.

  1. Enable the NGINX Ingress Controller
$ microk8s enable ingress dns
Infer repository core for addon ingress
Enabling Ingress
ingressclass.networking.k8s.io/public created
ingressclass.networking.k8s.io/nginx created
namespace/ingress created
serviceaccount/nginx-ingress-microk8s-serviceaccount created
clusterrole.rbac.authorization.k8s.io/nginx-ingress-microk8s-clusterrole created
  1. Map and expose our ingress controller with the already configured Metallb. We can do that using a Service. You can use your own ports here, but I’m using the default port for the HTTP because I assign a dedicated IP from my network.
apiVersion: v1
kind: Service
metadata:
  name: ingress
  namespace: ingress
  annotations:
    metallb.universe.tf/address-pool: default-addresspool
spec:
  ports:
  - port: 80
    targetPort: 80
    protocol: TCP
    name: http
  - port: 443
    targetPort: 443
    protocol: TCP
    name: https
  selector:
    name: nginx-ingress-microk8s
  type: LoadBalancer
  loadBalancerIP: 10.116.0.11

We apply the configuration.

$ kubectl apply -f nginx.yaml
service/ingress created
  1. Deployment file for the app.

We have the following components:

  • Deployment: Contains the specifications that our pod need to run inside the cluster.
  • Service: Publishes our pod ports inside the cluster to be accessible.
  • Ingress: Publishes our application using our nginx ingress controller and assigns a domain name and configuration rules.
apiVersion: v1
kind: Namespace
metadata:
  name: heimdall
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: heimdall
  namespace: heimdall
spec:
  selector:
    matchLabels:
      app: heimdall
  template:
    metadata:
      labels:
        app: heimdall
    spec:
      containers:
      - name: heimdall
        image: linuxserver/heimdall:latest
        ports:
        - containerPort: 80
          name: heimdallweb
          protocol: TCP
        env:
          - name: 'TZ'
            value: 'America/Lima'
          - name: 'PUID'
            value: '1000'
          - name: 'PGID'
            value: '1000'
        volumeMounts:
          - mountPath: '/config'
            name: heimdall-config
      volumes:
        - name: heimdall-config
          hostPath:
            path: /app/kubernetes/heimdall/config
---
apiVersion: v1
kind: Service
metadata:
  name: heimdallservice-web
  namespace: heimdall
spec:
  selector:
    app: heimdall
  ports:
  - port: 80
    targetPort: heimdallweb
    protocol: TCP
---
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: heimdall-ingress
  namespace: heimdall
  labels:
    app: home-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
  - host: "heimdall.mauricubo.com"
    http:
      paths:
      - pathType: Prefix
        path: /
        backend:
          service:
            name: heimdallservice-web
            port:
              number: 80

We apply the configuration. Of course you can change the parameters that we are using here, for example the domain name or the port.

$ kubectl apply -f heimdall.yaml
namespace/heimdall created
deployment.apps/heimdall created
service/heimdallservice-web created
ingress.networking.k8s.io/heimdall-ingress created

And finally… We test it…
